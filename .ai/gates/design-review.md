# 设计评审门禁

在**方案交付用户确认**或**进入实现前**通过。

## 检查清单

- [ ] **问题清晰**：能一句话说清要解决什么
- [ ] **范围明确**：有「纳入」与「不纳入」
- [ ] **已调研代码**：列出复用点，非凭空设计
- [ ] **方案可执行**：涉及层（web/backend/packages）与主要文件/接口已点名
- [ ] **有取舍**：多方案时已对比；单方案时已说明为何不选其他路径
- [ ] **可验证**：成功标准 + 验证方式（gates、冒烟、MCP 浏览器等）
- [ ] **安全**：无密钥进仓库、鉴权边界已考虑
- [ ] **产品对齐**：与本仓库定位一致（文档 AI 平台；LLM 生成走 SSE）

## 可豁免

- 用户明确「就改 X」且单文件、意图无歧义
- 纯文案 / 文档修正，无行为变更

## 未通过时

回到 [requirements-design workflow](../workflows/requirements-design.md) 补澄清或补调研，**不要**直接写大段业务代码。
