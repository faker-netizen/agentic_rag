# 规格偏差处理

用户表示「不是我要的」→ **先分类、更新落盘文件、再改代码**。

## 触发

实现后验收不通过；或设计阶段用户驳回。

## 分类决策树

```text
用户反馈「不对」
    │
    ├─ 档位定错了？ ──→ 回到 tier-classification，改 T0–T3，迁移/新建对应目录
    │
    ├─ T0：期望行为理解错？ ──→ 可能升档 T1+，转 planning
    │
    ├─ T1/T2/T3：目标/范围错了？ ──→ 修订 01-requirements（T2/T3）或升档重写 lite
    │
    ├─ AC / 交互 / 边界不足？ ──→ T1 修订 01-change-lite；T2/T3 修订 02-solution-design
    │
    └─ AC 已清，实现未达？ ──→ 留实现阶段；T0 更新 00-bugfix 验证项
```

## 按档位修订文件

| 档位 | 回流目标 | 修订文件 |
|------|----------|----------|
| T0 | 理解错 / 非 bug | 升档 + `planning/` |
| T0 | 修完仍不对 | `fixes/.../00-bugfix.md` |
| T1 | 需求/范围 | `01-change-lite.md` → 待确认 |
| T1 | 实为 T2 | 升档，补 `01`+`02` |
| T2/T3 | 需求 | `01-requirements.md` |
| T2/T3 | AC/方案 | `02-solution-design.md` |
| 任意 | 实现 | 对照已确认 AC，最小 diff |

确认记录改回「待确认」→ 过对应 gate → **等用户确认** → 再继续。

## 禁止

连续 patch 碰运气；只改对话不落盘。
